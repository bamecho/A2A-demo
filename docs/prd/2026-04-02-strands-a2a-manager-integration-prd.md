# PRD: Strands A2A Server 与 Agent Manager 集成

## Problem Statement

当前系统已经基于 `strands-agent + FastAPI` 提供面向用户的 API，请求进入后由内部 `manager` 根据 `user_id` 创建或复用 agent 来处理用户消息。这套能力目前主要服务于本地或自有 API 入口，缺少一个标准化的远程 agent-to-agent 接入面。

系统现在需要新增一个作为服务端的标准 A2A 接口，使内部受信任服务能够通过标准 A2A SDK 调用现有 agent 能力，并获得稳定的流式响应。难点在于：Strands 官方 A2A server 的默认建模更接近“服务绑定固定 agent”，而当前系统的核心建模是“服务请求绑定用户上下文，由 manager 决定复用哪个 agent”。如果不解决这两种生命周期模型之间的错位，就会导致 agent 复用语义丢失、身份边界混乱、并发行为不稳定，或者需要对现有 manager 做过度侵入式改造。

## Solution

新增一个挂载在现有 FastAPI 应用中的标准 A2A server 入口，对外兼容 Strands 官方 A2A SDK，对内继续复用现有 `manager` 作为 agent 生命周期与会话复用的唯一所有者。

该方案的核心是引入一层面向 A2A 的桥接执行器：

- 在 HTTP 鉴权层解析可信 `user_id` 与请求上下文
- 在 A2A 执行阶段按请求动态调用 `manager` 获取或复用该用户 agent
- 将 A2A 输入映射到现有 agent 输入模型
- 将 agent 的流式输出映射为标准 A2A streaming response
- 对同一 `user_id` 的并发请求采用串行化或明确返回 busy 的策略，保证与 Strands agent 的并发限制一致

这样可以在不重构现有 manager 的前提下，为内部服务提供标准化远程接入能力，同时保留当前用户 API 所依赖的 agent 复用与上下文连续性。

## User Stories

1. 作为内部平台服务，我希望通过标准 A2A SDK 调用现有 agent 服务，这样我不需要为该系统编写私有协议适配层。
2. 作为内部平台服务，我希望远程 A2A 调用能够拿到流式输出，这样我可以在上游系统中实时展示或消费 agent 结果。
3. 作为内部平台服务，我希望远程 A2A 能力面与现有用户 API 基本等价，这样我不需要维护一套功能明显缩水的远程接入路径。
4. 作为平台集成人员，我希望远程 A2A 请求在进入执行链前就完成可信身份解析，这样系统不会依赖客户端在请求体中自报 `user_id`。
5. 作为平台集成人员，我希望系统能够在 HTTP 层提取并校验用户上下文，这样鉴权、审计和限流边界保持清晰。
6. 作为系统架构师，我希望 A2A 层不直接拥有 agent 生命周期，这样现有 manager 的职责边界不会被破坏。
7. 作为系统架构师，我希望 manager 继续决定 agent 的创建、复用和保活策略，这样远程接入不会改变既有用户行为。
8. 作为系统维护者，我希望 A2A server 可以直接挂载到现有 FastAPI 服务中，这样部署、日志、鉴权和配置都能复用现有基础设施。
9. 作为系统维护者，我希望远程 A2A 请求和现有 API 请求共享同一套身份上下文模型，这样链路追踪和故障排查更一致。
10. 作为后端开发者，我希望 A2A 协议适配逻辑与 manager 对接逻辑解耦，这样后续替换协议层或内部 manager 时改动面可控。
11. 作为后端开发者，我希望 A2A 输入映射和输出映射是独立模块，这样协议转换规则可以单独测试。
12. 作为后端开发者，我希望第一版只支持文本输入和文本流式输出，这样可以快速交付最小可用闭环并降低协议复杂度。
13. 作为后端开发者，我希望非文本 parts、断线恢复和持久任务状态先不纳入第一版，这样范围足够收敛。
14. 作为产品负责人，我希望内部服务可以在不理解现有私有 API 协议的前提下完成接入，这样跨团队协作成本更低。
15. 作为产品负责人，我希望第一版重点验证“内部服务能否通过标准 A2A SDK 稳定接入”，这样可以尽快确认协议路线正确。
16. 作为 QA，我希望系统对认证失败、缺失用户上下文、manager 获取 agent 失败、agent 执行异常等场景给出稳定错误响应，这样客户端和平台都能做一致处理。
17. 作为 QA，我希望系统对同一用户的并发远程请求给出确定性行为，这样不会出现偶发竞态或上下文污染。
18. 作为内部服务调用方，我希望当同一用户已有活跃执行时，系统要么串行排队、要么明确返回 busy，这样我可以做可预期的重试或降级。
19. 作为 SRE，我希望每次 A2A 调用都带有可追踪的 `request_id` 和用户上下文，这样异常可以跨网关、FastAPI、manager 和模型调用链路追踪。
20. 作为 SRE，我希望系统对客户端断开连接时有清晰的资源释放策略，这样不会造成明显的资源泄漏。
21. 作为安全负责人，我希望内部服务调用 A2A server 仍然经过统一鉴权，这样不会因为协议标准化而弱化边界控制。
22. 作为安全负责人，我希望对外错误响应不泄漏内部 traceback 和 manager 实现细节，这样系统不会暴露不必要的内部信息。
23. 作为研发负责人，我希望第一版尽量少改现有 manager，这样可以降低回归风险并缩短交付时间。
24. 作为研发负责人，我希望后续如果要增加任务恢复、富媒体或更完整的 A2A 生命周期，也能在当前设计上演进，而不是推翻重做。
25. 作为测试开发者，我希望关键协议桥接层既有集成测试也有针对 mapper/translator/provider 的单元测试，这样既能验证端到端行为，也能快速定位问题。
26. 作为内部服务调用方，我希望连续两次对同一用户发起远程请求时，仍能命中同一类上下文复用语义，这样远程接入与现有 API 表现一致。
27. 作为内部服务调用方，我希望不同用户的远程请求可以并行执行，这样系统不会因为单用户保护策略而整体吞吐退化。
28. 作为平台工程师，我希望 A2A 层可以明确声明第一版支持边界，这样上游团队不会误用尚未实现的生命周期和多模态能力。

## Implementation Decisions

- 远程能力只面向内部受信任服务开放，第一版不按开放平台场景设计。
- A2A server 挂载在现有 FastAPI 应用内，而不是单独运行一套服务。
- 系统将继续使用 Strands 官方 A2A server 抽象作为协议外壳，但不会使用“固定绑定单个 agent”的默认执行模型。
- 需要引入一层自定义执行桥接，使每个 A2A 请求在执行时动态解析当前用户上下文，并通过内部 manager 获取或复用 agent。
- 可信 `user_id` 只从 HTTP 鉴权链路或受信任网关注入上下文中获取，不从 A2A 请求体 metadata 中信任性提取。
- 现有 manager 仍然是 agent 生命周期、上下文复用和保活策略的唯一权威来源。
- A2A 协议层只负责请求接入、输入转换、输出流转换和错误映射，不承担 agent 生命周期管理。
- 第一版仅支持文本输入与文本流式输出，能力面与现有用户 API 尽量保持等价，但不承诺完整透传所有内部工具事件。
- 需要定义稳定的输入映射模块，将 A2A message/parts 转换为现有 agent 可消费的输入结构。
- 需要定义稳定的事件翻译模块，将 Strands agent 的流式输出转换为 A2A 可消费的流式事件。
- 第一版不实现持久 `task_store`、断线恢复或长任务查询，远程调用以实时流式消费为中心。
- 同一 `user_id` 的并发请求需要采用确定性策略，第一版默认设计为串行化或直接返回 busy，以符合 Strands agent 对单实例并发的限制。
- 不同 `user_id` 的请求应允许并行执行，前提是它们对应不同的 agent 实例或独立执行上下文。
- 错误处理需要统一覆盖认证失败、上下文缺失、agent 获取失败、并发冲突、执行异常、客户端断连等主要场景。
- 所有远程请求应具备统一的链路追踪字段，例如 `request_id`、用户上下文和必要的审计标签。
- 第一版的成功标准优先级为：内部服务可以使用标准 A2A SDK 直接接入并拿到稳定流式结果。

## Testing Decisions

- 好的测试应只验证外部可观察行为，而不是绑定实现细节；重点验证请求如何进入系统、系统如何选择 agent、以及客户端最终接收到什么流式结果或错误。
- 测试策略采用“集成测试为主，关键边界单元测试补齐”的方式。
- 需要覆盖的集成测试包括：标准 A2A 请求成功流式返回、同一用户连续请求复用语义、同一用户并发请求 busy/串行行为、不同用户并发互不污染、认证失败拦截、manager 异常映射、agent 异常映射、客户端断开时的资源清理。
- 需要覆盖的单元测试包括：身份上下文解析、A2A 输入映射、事件翻译、并发策略判断、错误映射规则、manager provider 适配逻辑。
- 测试中的断言应聚焦于协议层输出、状态码、事件序列、错误码和上下文隔离，而不是内部私有方法调用次数。
- 若代码库中已有 FastAPI 集成测试、流式响应测试或 adapter/mapper/translator 风格的测试，应优先复用相同测试模式和断言风格；若尚未存在，则以端到端协议验证为主建立基线。

## Out of Scope

- 面向外部第三方或开放平台的 A2A 暴露模式
- 持久任务状态追踪、任务恢复、轮询查询和断线续传
- 真正的远程取消语义与底层执行强制中断
- 多模态输入输出、非文本 parts、附件和富媒体 artifact
- 将所有内部工具调用细节无损透传为标准 A2A 事件
- 为第一版重构或替换现有 manager 的核心职责
- 与现有用户 API 无关的额外 agent 能力扩展

## Further Notes

- 第一版的关键是用最小侵入方式证明“标准 A2A SDK -> FastAPI A2A server -> manager 复用 agent -> 流式返回”这条链路可行且稳定。
- 设计中应预留向更完整 A2A 生命周期扩展的边界，但不应为尚未需要的恢复、持久化和多模态能力提前付出过高复杂度成本。
- 如果后续要支持更完整的任务模型，优先考虑在 A2A 协议层和 manager 之间增加独立的 task/session adapter，而不是直接把任务状态逻辑塞进 manager 或协议 handler。
