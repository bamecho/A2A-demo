# Lessons

## 2026-04-02

- 当用户明确更改执行顺序或新增前置要求时，先调整执行计划并优先完成该前置步骤，再继续后续 phase/task。
- 在承诺使用 `git worktree` 前，先确认当前工作目录是否是受 git 跟踪的项目根或受跟踪子树；如果不是，先显式说明约束，避免给出不可执行的隔离方案。
- 当执行多 phase 计划时，每完成一个 phase 都必须显式停在用户 review 闸门，先给出该 phase 的结果、验证、风险和当前状态，再决定是否继续下一 phase。
- 如果存在 plan 文档，阶段性 review 和验证结果必须写回 plan 或相邻执行文档，不能只留在对话里后直接继续开发。
- 当工作已跨越多个 phase 且修改叠加在同一文件时，不要假装可以无成本拆成完美历史；应明确说明约束，并选择当前可安全执行的最小原子提交方案。
- 对应 spec 文档必须同步更新状态与 review，而不只是更新 plan；代码完成与文档回写应拆成独立提交。
